Document: BOPIS Playwright CI/CD Implementation Manual

Date: February 16, 2026

Audience: QA Engineers, Developers, Managers, and New Team Members

1. Executive Summary

This manual explains how BOPIS test automation is implemented using Playwright and how CI/CD is executed via GitHub Actions. It documents what has been completed, how the pipeline works end to end, what application flows are covered, and what operational checks are required to keep CI stable.

2. Objectives

  • Run BOPIS regression automatically on code changes and release tags.
  • Detect UI/functional regressions early.
  • Produce consolidated test reports for release decisions.
  • Securely manage credentials and session state.
  • Reduce manual execution and increase release confidence.

3. Scope

In Scope

  • Playwright framework configuration and test execution.
  • GitHub Actions workflow for shard-based execution and report merge.
  • Authentication setup strategy (single setup auth + storage state reuse).
  • BOPIS order workflows across Open, Packed, and Completed stages.
  • Negative and lifecycle scenarios.

Out of Scope

  • Application feature development.
  • Infrastructure changes outside GitHub Actions runner context.

5. Technical Implementation (What We Built)

5.1 Repository Components

  • Workflow: .github/workflows/playwright-on-tag.yml
  • Playwright config: playwright.config.mjs
  • Auth setup spec: tests/specs/auth.setup.js
  • Main regression specs: tests/specs/order/*.spec.js, tests/specs/login-flow.spec.js

5.2 CI Workflow Design

  1. Checkout repository.
  2. Setup Node.js 20 and npm cache.
  3. Generate runtime .env from .env.example using GitHub secrets.
  4. Install npm dependencies.
  5. Install Playwright browsers with dependencies.
  6. Run tests in 3 shards with reporters: blob + JSON.
  7. Upload per-shard artifacts (blob and JSON).
  8. Merge blob reports into one HTML report in a merge job.

5.3 Playwright Execution Strategy

  • CI retries enabled only in CI (retries: 1).
  • Traces and videos retained only for failed tests.
  • Project split:
    • setup: creates authenticated storage state once.
    • chromium: runs business tests with saved auth session.
    • login-flow: validates login scenarios independently.

6. Security and Configuration Controls

6.1 Required GitHub Secrets

  • CURRENT_APP_URL
  • OMS_NAME
  • USERNAME
  • PASSWORD

Critical Rule: All four secrets must belong to the same environment family (example: dev URL + dev OMS + dev user). Mixed environments cause login redirect failures in CI.

6.2 Git Hygiene

The following are excluded from source control to avoid sensitive data exposure:

  • .env
  • auth*.json
  • playwright/.auth/
  • playwright-report/
  • test-results/
  • node_modules/

7. Application Coverage Implemented

7.1 Login and App Entry

  • Launchpad entry and BOPIS app selection.
  • Positive login flow and login validation scenarios.

7.2 BOPIS Orders Coverage

  • Open tab actions: assign picker, pack, print picklist.
  • Packed tab actions: handover, print packing slip, gift card activation.
  • Completed tab validations and order-detail checks.
  • Reject and cancel flows for single and multiple item scenarios.
  • Lifecycle flow: Open -> Packed -> Completed.
  • Negative scenarios (empty states, missing actions, persistence checks).

8. Known Failure Pattern and Root Cause

Observed CI error: Login stays on URL like /login?oms=...&token=... and does not reach /tabs/orders.

Primary cause: Domain/environment mismatch in secrets (for example, prod URL with dev OMS/credentials).

Why local works: Local .env is usually aligned (same env domain + OMS + user), so session establishment succeeds.

9. Onboarding Guide for New Team Members

  1. Clone repository and run npm ci.
  2. Create local .env from .env.example.
  3. Set CURRENT_APP_URL, OMS_NAME, USERNAME, PASSWORD to the same environment.
  4. Run smoke test locally: npx playwright test tests/specs/login-flow.spec.js --headed.
  5. Run a target suite: npx playwright test tests/specs/order/handover.spec.js --headed.
  6. Review reports in playwright-report/.
  7. Before raising issue, collect trace/video from failed test artifacts.

11. Definition of Done (CI Stable)

  • Auth setup passes consistently in CI.
  • Regression suite runs end-to-end across shards.
  • No sensitive files are committed.
  • Merged HTML report is available for every CI run.
  • No environment mismatch between URL, OMS, and credentials.
